IFNOTUS — VPS Production Deployment Readiness Audit
====================================================
Date: 2026-07-06
Scope: Production deployment risk (not feature completeness)
Target: Frontend static build behind Nginx + backend on same VPS, HTTPS, /api/ reverse proxy

================================================================================
A. SAFE-TO-DEPLOY VERDICT
================================================================================

Deployable with fixes

The codebase can work on a VPS with Nginx serving frontend/dist and proxying
/api/ to uvicorn on 127.0.0.1:8000, IF production build uses relative API URLs
(/api/v1) and backend env is hardened. Deploying as-is (dev Vite server, baked-in
localhost API URL, default admin, dev port monitoring) is NOT safe.

================================================================================
B. CRITICAL BLOCKERS
================================================================================

1. PRODUCTION BUILD CAN BAKE IN localhost API URL
   - docker-compose.yml sets VITE_API_BASE_URL=http://localhost:8000/api/v1 for frontend
   - frontend/.env.example recommends the same
   - If npm run build runs with that env var, browser calls http://localhost:8000 from
     users' machines → broken API, mixed content on HTTPS pages
   - Fix: Build with VITE_API_BASE_URL=/api/v1 or unset (client.ts defaults to /api/v1)

2. FRONTEND DOCKERFILE / COMPOSE RUN DEV SERVER, NOT STATIC BUILD
   - frontend/Dockerfile: CMD npm run dev -- --host (port 5173)
   - docker-compose frontend service maps 5173 and mounts source
   - Production must serve frontend/dist via Nginx, not Vite dev

3. DEFAULT ADMIN CREDENTIALS
   - Makefile seed-admin: admin / admin123
   - backend/app/cli/seed_admin.py defaults: admin / admin123
   - Must use strong password in production; disable or protect seed CLI

4. DEV SECRETS IN .env.example
   - SECRET_KEY=change-me-to-a-random-string-at-least-32-chars-long
   - ENVIRONMENT=development, DEBUG=true, RELOAD=true
   - Deploying without overriding these is unsafe

5. MONITORING EXPECTS DEV PORT 5173 IN PRODUCTION
   - backend/app/core/config.py: monitoring_expected_ports default [8000, 5173, 5432, 6379]
   - backend/app/services/monitoring/service.py: hardcoded port_labels includes 5173
   - Static Nginx frontend does not listen on 5173 → constant false "Frontend not listening" alerts

6. APPLICATION YAML paths.root IS CWD-RELATIVE (AMBIGUOUS ON VPS)
   - applications/ifnotus-api.yaml: paths.root: "."
   - applications/ifnotus-frontend.yaml: paths.root: "../frontend"
   - ApplicationDefinition.root_path = Path(string) with no anchor to repo root or YAML dir
   - DeploymentHistoryReader uses app.root_path / ".ifnotus" / "deployments"
   - Wrong CWD in systemd unit → broken deployment history, git/nginx/ssl readers

7. MULTI-WORKER BACKEND BREAKS IN-MEMORY MONITORING STATE
   - MonitoringService, CollectorCache, MetricsHistory are in-process singletons
   - uvicorn --workers N > 1 → inconsistent charts/cache per worker
   - Must run WORKERS=1 until history/cache shared via Redis

8. NO ACCESS TOKEN REFRESH ON FRONTEND
   - refresh_token stored in localStorage but never used
   - ACCESS_TOKEN_EXPIRE_MINUTES=30 → users forced to re-login every 30 minutes
   - Not a deploy blocker but production UX/security gap

9. BACKEND LISTENS ON 0.0.0.0:8000 BY DEFAULT
   - config.py host=0.0.0.0 exposes API directly if port not firewalled
   - Behind Nginx, bind 127.0.0.1:8000 and block public 8000

10. MISSING .env.production FOR FRONTEND
    - No frontend/.env.production committed or documented for build-time injection

  Fix: Add file with VITE_API_BASE_URL=/api/v1

frontend/Dockerfile
  Issue: Runs npm run dev on 5173

  Fix: Not for production VPS; use compose only for local dev

D. REQUIRED BACKEND FIXES BEFORE DEPLOYMENT
================================================================================

backend/app/core/config.py
  Issue: cors_origins default ["http://localhost:5173"]
  Fix: Set CORS_ORIGINS=https://your-domain.com in production IF frontend ever calls API
       cross-origin. With same-origin Nginx /api/ proxy, browser same-origin requests
       don't need CORS, but keep origin list correct if split later.

  Issue: redis_url default redis://localhost:6379/0
  Fix: Set REDIS_URL to VPS Redis socket/host in production .env

  Issue: monitoring_expected_ports includes 5173
  Fix: MONITORING_EXPECTED_PORTS=8000,5432,6379 (or 80,443 if monitoring edge ports)

backend/app/services/monitoring/service.py (lines ~374-387)
  Issue: Hardcoded port_labels {8000, 5173, 5432, 6379}
  Fix: Derive labels from config or application registry; remove 5173 for static frontend

.env / .env.example
  Issue: ENVIRONMENT=development, DEBUG=true, RELOAD=true, weak SECRET_KEY
  Fix: ENVIRONMENT=production, DEBUG=false, RELOAD=false, WORKERS=1 (until Redis history),
       cryptographically random SECRET_KEY >= 32 chars

backend/app/main.py
  Issue: Docs disabled when is_production — good
  Issue: No TrustedHostMiddleware
  Fix: Add TrustedHostMiddleware with allowed production hostnames (optional but recommended)

backend/app/cli/seed_admin.py + Makefile
  Issue: Default admin123 password
  Fix: Require explicit password arg in prod; never run seed with defaults on VPS

applications/ifnotus-api.yaml
  Issue: paths.root: "." — resolves relative to process CWD
  Fix: Use absolute path e.g. /opt/ifnotus/backend or resolve paths relative to YAML file

applications/ifnotus-api.yaml + ifnotus-frontend.yaml
  Issue: nginx.server_name: localhost, ssl.domain: localhost
  Fix: Set production domain in YAML for accurate nginx/ssl monitoring

backend/Dockerfile
  Issue: Single worker uvicorn OK; dev dependencies installed (.[dev])
  Fix: Production image should pip install -e . without dev extras; no --reload

Makefile backend target
  Issue: --reload flag for local dev
  Fix: Production systemd unit must not use --reload

No rate limiting on /auth/login
  Fix: Add rate limiting at Nginx or app layer before public exposure

Logout endpoint
  Issue: Token blacklist not implemented (auth.py comment)
  Fix: Acceptable for MVP but stolen tokens valid until expiry

Monitoring collectors on Linux VPS
  Verify: psutil net_connections may need CAP_NET_ADMIN or run as appropriate user
  Verify: MONITORING_LOG_PATHS=/var/log/syslog exists on target OS

E. REQUIRED NGINX / VPS ASSUMPTIONS
================================================================================

Production server must provide:

1. Static frontend root
   - Path: e.g. /var/www/ifnotus/dist (output of cd frontend && npm run build)
   - NOT Vite dev server on 5173

2. Backend upstream (internal only)
   - uvicorn app.main:app --host 127.0.0.1 --port 8000
   - Firewall: deny public access to 8000; only Nginx → backend

3. /api/ reverse proxy
   - Browser requests: https://your-domain.com/api/v1/...
   - Nginx must proxy to http://127.0.0.1:8000/api/ (preserve /api/v1 prefix)
   Example:
     location /api/ {
       proxy_pass http://127.0.0.1:8000/api/;
       proxy_http_version 1.1;
       proxy_set_header Host $host;
       proxy_set_header X-Real-IP $remote_addr;
       proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
       proxy_set_header X-Forwarded-Proto $scheme;
     }

4. WebSocket proxy
   - NOT REQUIRED — no WebSocket/SSE usage in frontend/src (HTTP polling only)

5. SPA fallback (required)
   - Router uses createWebHistory (HTML5 history mode)
   - Direct refresh of /monitoring, /servers, /settings must fall back to index.html
   Example:
     location / {
       try_files $uri $uri/ /index.html;
     }

6. HTTPS
   - Terminate TLS at Nginx
   - Frontend must not call http:// API URLs (mixed content)

7. Headers / cookies
   - Auth is Bearer token in localStorage (not cookies)
   - withCredentials NOT used — no cookie auth config needed
   - CORS mostly irrelevant for same-origin /api/ proxy setup

8. PostgreSQL + Redis
   - Running on VPS or managed service; DATABASE_URL and REDIS_URL in backend .env

9. systemd (recommended)
   - WorkingDirectory=/opt/ifnotus (repo root) so applications/ and paths resolve consistently
   - Run alembic upgrade head before first start
   - Do not expose Makefile seed-admin defaults in prod

G. PERFORMANCE / POLLING RISK NOTES
================================================================================

Poll interval: REALTIME_POLL_MS = 5000 (frontend/src/config/polling.ts)

Per authenticated user, approximate request rate:

  Dashboard page (useDashboard):
    Every 5s: GET /dashboard (HEAVY — multiple collectors)
              GET /health, GET /health/ready
              GET /applications
              Up to 6x GET /applications/{id}/deployments
    ≈ 9+ requests / 5s ≈ 108+ req/min
    Plus AppTopBar: GET /alerts every 5s (+12 req/min)

  Monitoring page:
    6 independent usePolling hooks each firing every 5s
    Includes another GET /dashboard (heavy duplicate)
    ≈ 72 req/min minimum

  Servers page: 3 polls × 12/min = 36 req/min
  Settings page: 3 polls × 12/min = 36 req/min

Backend collector cache TTL: 15s (monitoring_cache_ttl_seconds)
  - Mitigates duplicate collector work within 15s window per worker
  - get_dashboard() still orchestrates many collectors each call

Verdict:
  - Acceptable for 1–3 admin operators on a small VPS
  - Risky for many concurrent users or small VPS (1 CPU): dashboard endpoint runs
    CPU/memory/disk/network/processes/ports/services collectors
  - Monitoring page double-fetches dashboard + metrics — wasteful
  - Intervals DO clear on component unmount (usePolling, useDashboard, notifications)
  - No WebSocket/SSE today; polling is simpler but heavier than push model
  - Recommend before scale: increase poll interval to 10–15s in production, dedupe
    MonitoringView dashboard+metrics calls, add If-None-Match or server-side throttle
